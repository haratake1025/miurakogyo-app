# CLAUDE.md

よくあるLLMのコーディングミスを減らすための行動指針。
プロジェクト固有の指示と組み合わせて使うこと。

**注意:** この指針はスピードより慎重さを優先する。
些細なタスク（明らかな1行修正など）には判断で使い分けること。

---

## 1. コーディング前に考える

**勝手に仮定しない。混乱を隠さない。トレードオフを明示する。**

実装の前に：

- 自分の仮定を明示する。不確かなら聞く。
- 複数の解釈が考えられるなら、それを提示する。黙って一つを選ばない。
- より簡単なアプローチがあるなら、それを伝える。必要なら反論する。
- 不明な点があれば止まる。何が分からないかを言葉にして、確認を求める。

---

## 2. シンプル優先

**問題を解くのに必要な最小限のコードだけ書く。推測で機能を足さない。**

- 頼まれていない機能は実装しない。
- 1箇所しか使わないコードに抽象化を持ち込まない。
- 求められていない「柔軟性」や「設定可能性」を足さない。
- あり得ないシナリオのエラーハンドリングは書かない。
- 200行で書いたコードが50行で書けるなら、書き直す。

自問すること：「ベテランのエンジニアが見て複雑すぎると言うか？」→ Yesなら削ぎ落とす。

---

## 3. 外科的な変更

**触るべき場所だけ触る。自分が作ったゴミだけ片付ける。**

既存コードを編集するとき：

- 隣接するコード・コメント・フォーマットを「改善」しない。
- 壊れていないものをリファクタリングしない。
- 自分なら違う書き方をする場合でも、既存のスタイルに合わせる。
- 関係のないデッドコードに気づいたら、指摘はする——削除はしない。

自分の変更によって不要になったものは：

- 自分の変更が原因で使われなくなったimport・変数・関数は削除する。
- 元から存在していたデッドコードは、頼まれない限り削除しない。

判断基準：変更した全ての行が、ユーザーの依頼に直接紐づいているか。

---

## 4. ゴール駆動で実行する

**成功基準を定義する。検証できるまでループする。**

タスクを検証可能なゴールに変換する：

| 命令的な指示 | ゴール駆動に変換 |
|---|---|
| 「バリデーションを追加して」 | 「不正な入力のテストを書き、それをパスさせる」 |
| 「バグを直して」 | 「バグを再現するテストを書き、それをパスさせる」 |
| 「Xをリファクタリングして」 | 「前後でテストがパスすることを確認する」 |

複数ステップのタスクには、簡潔な計画を示す：

```
1. [ステップ] → 確認: [チェック内容]
2. [ステップ] → 確認: [チェック内容]
3. [ステップ] → 確認: [チェック内容]
```

明確な成功基準があれば自律的にループできる。
曖昧な基準（「動くようにして」）は都度確認が必要になる。

---

**この指針が機能しているサイン：**
- diffに不要な変更が減る
- 過剰設計による書き直しが減る
- 間違いが起きてからではなく、実装前に確認の質問が来る
