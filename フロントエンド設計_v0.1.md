# フロントエンド設計書（v0.1）― 出面・石綿記録システム

本体仕様書・バックエンド設計・CBO API連携仕様 を前提とした画面/コンポーネント設計。**PC優先**。セル操作は **A案（インライン編集）**。

---

## 1\. 技術・全体方針

- Next.js（App Router）/ React / TypeScript、Tailwind CSS。  
- データ取得は **フロント → Next.js APIルート**（バックエンド設計§3）のみ。秘密情報は保持しない。  
- サーバ状態管理に **TanStack Query**（キャッシュ＋楽観更新＋invalidate）。  
- 祝日判定はフロントの祝日ライブラリ（例: holiday-jp 相当）で土日祝の色分け。  
- 既存v2のサイドナビ（現場一覧／マスタ管理／セットアップ）とURL（`/sites/{id}/...`）の良い部分は踏襲。

## 5\. 出面表グリッド（A案・インライン編集）

- **1セル \= 1 daily\_report**（site × worker × date）。  
- **空セル**: クリックで新規作成（`local_new`）。既定値 \= 昼勤(105360)/健康状態〇良好(106556)。  
- **入力済セル**: 在席を表示（例: ●、夜勤は●夜、残業ありは ●+2h）。クリックで**インラインポップオーバー**編集:  
  - 昼勤/夜勤（トグル 105360/105361）  
  - 残業 over\_hour（数値）  
  - 石綿現場なら 作業内容（単一）・健康状態（このセルから石綿項目も編集可＝両ビュー連動）  
  - 削除（ガード: 確認ダイアログ）  
- **sync\_status をセルに重畳**（凡例必須）:  
  - `local_new` … 例: `!`／点線枠  
  - `local_edited` … `▲`／オレンジ点  
  - `synced` … 通常表示  
  - `conflict` … `✕`／赤  
- **キーボード操作（PC前提）**: 矢印でセル移動、Enterで編集確定、Tabで次セル。連続入力を高速化。  
- 任意: 行/列まとめて昼勤などを適用するクイック一括入力。

## 6\. 石綿作業従事者作業記録グリッド

- 上期(1–15)/下期(16–末) 切替、行=作業者・列=日。  
- セル \= **作業内容番号①〜⑧（単一）＋健康状態（〇/△/✕）**。インライン編集は作業内容 single select＋健康状態。  
- 凡例（作業内容①〜⑧・健康状態）を上部に常設。`is_corrected` のセルに「修正済」バッジ。  
- ヘッダ: 管轄工事会社(client\_name)/工事名称(name)/工事期間(period)/現場責任者(manager\_name)。  
- 「出面集計Excel出力」「印刷（石綿記録 法定様式）」。  
- 出面表と同一データなので、どちらのビューで編集しても相互に反映。

## 7\. 作業者一括追加モーダル（AddWorkerModal）

1. **会社名を選択**: 三浦興業（社員マスタ）/ 協力会社一覧（workers の partner 会社）。  
2. **氏名を複数チェック**（その会社の workers、氏名検索ボックスあり）。既に表に居る作業者は除外/グレーアウト。  
3. **\[追加\]** → 選択分を当該現場のグリッドに**行追加**（＝表示対象への追加。マスタ新規登録ではない）。

マスタ自体の追加・編集はCBO側が正。アプリからは新規登録しない。

## 8\. 同期フロー（UI挙動）

- **CBOから取込（pull）**: `/api/sync/pull/masters` ＋ `/api/sync/pull/reports`（siteId・表示期間）。完了トーストに差分件数。`conflict` 発生時はセル＝✕＋`/sync` で解決導線。  
- **CBOへ反映（push）**: 未同期（local\_new/local\_edited）件数をバッジ表示。押下で `/api/sync/push`。CBOレート対策のスロットリング中はプログレス表示。成功でセルが `synced` 表示へ。  
- **ConflictResolver**（/sync）: 競合行ごとに「CBO版 / アプリ版」を並べて選択（CBO採用=上書き取込／アプリ採用=再push）。  
- TanStack Query: 編集・push は楽観更新→確定 or ロールバック、関連クエリを invalidate。

---

## 9\. 状態管理

- queryKeys: `['sites']` / `['site', siteId, 'reports', month]` / `['workers', company]`。  
- mutations: createReport / updateReport / deleteReport / pushSync / pullSync → 成功時に該当キーを invalidate。  
- ローカル編集は即サーバ（Next API → Supabase）へ保存し `sync_status` を更新。オフライン下書きは初版スコープ外（将来）。
