# AIソロプレナー起業ネタ調査レポート

**作成日**: 2026-07-10
**調査方法**: AIエージェント5体による分業（市場調査3体の並列実行 → アイデア生成1体 → レッドチーム批評1体 → 統合）。市場調査・競合確認あわせて60回超のウェブ検索を実施。

---

## エグゼクティブサマリー

- **最有力は「日本の建設下請け向け 作業員マスターデータハブ（安全書類×出面×資格管理の統合）」**。12案を6軸で評価した結果、唯一「ドメイン知識・既存開発資産・営業チャネル」の3つが揃う案だった。
- 調査で「未充足ニーズ」とされた領域の**大半（12案中10案）に既に直接競合が出荷済み**だった。ペインの実在と競合の不在は別物であり、これがAI起業アイデア検討の最大の落とし穴。
- 米国市場向けの案は市場としては魅力的だが、**日本在住・英語営業チャネルなしのソロにとって「作れるか」ではなく「売れるか」で全滅に近い**。ソロ起業の死因第1位は開発ではなく顧客獲得。
- 価格戦略の鉄則: **$50/月未満のAI製品は年間解約率77%（GRR 23%）で構造的に死ぬ。$250/月以上のB2BはGRR 70%**で通常のSaaS並み。単価の低いB2C AIは避けるべき（ChartMogul調査）。

## 第1部: 市場調査サマリー

### 1-1. 成功事例から見えた「一人で勝つ」パターン

2025〜2026年の収益公開事例（検証性の高いもの）:

| 事例 | 内容 | 収益 |
|---|---|---|
| Pieter Levels（Photo AI 他） | AI写真生成ポートフォリオ | 年商約$3.1M・従業員ゼロ |
| Danny Postma（HeadshotPro） | AI証明写真、買い切り課金×SEO | 月商約$300K |
| Marc Lou（ShipFast 他） | ボイラープレート＋教育 | 2025年 $1.03M（本人公開） |
| Tony Dinh（TypingMind） | LLMチャットUI→B2B展開 | 月商$130-160K |
| 匿名（理学療法クリニック向け事前承認エージェント） | 地味なバーティカルAIエージェント | 14ヶ月でMRR $41K |
| 大野克也（rimad・日本） | ひとり法人、創業1年で月商1,000万円 | 2億円超でEXIT |

**共通パターン**:
1. ディストリビューション（顧客への到達手段）が製品より先 — build in public / SEO / 業界人脈
2. 圧倒的なスピードと数打ち（Levelsは70回失敗の末にPhoto AI）
3. 課金はシンプルに。スケール時のみB2B化
4. **自分の専門領域から始める — ドメイン知識が最も安価な参入障壁**
5. 現実の分布: マイクロSaaSの中央値はMRR $4.2K。「一人で$10K MRR超」は上位10〜15%

### 1-2. 未充足ニーズ調査の核心

2025〜2026年の一貫したパターンは、**「非構造ドキュメント（メール・PDF・FAX・手書き・写真）と基幹システムの間の転記労働」がLLMで初めて解ける領域になった**こと。有望とされた領域の共通項:

- (a) 人件費の直接代替でROIが説明できる
- (b) 規制・法令が購買のドライバー（例: 建設業の2024年問題、5年ごとの許可更新）
- (c) 非構造データ処理というLLM適性
- (d) 顧客が中小・分散型でソロの営業が届く

具体的に定量ペインが確認された領域: 米国保険代理店のACORD/COI処理（CSRが週34時間を手作業入力に消費）、建設サブコンの書類業務（年間$18万〜45万の間接費）、日本の建設安全書類（施工計画書作成5日以上・書類AI化で年200万円削減試算）、士業の申請書類（AI利用率66%だが「汎用AIでは手間が増えた」層が存在）など。

### 1-3. 競合・リスク調査の核心

- **AIツールの墓場**: 死亡登録1,774件、最多は汎用ライティング（30%）。Jasper（売上半減）、Tome（2,000万ユーザーでもARR $4M未満→事業終了）が象徴。
- **Sherlockingリスク**: OpenAIのファイルアップロード機能追加で「ChatGPT for PDF」系が一夜で陳腐化。**「モデルの進化が自社製品を良くする位置に立て。不要にする位置に立つな」**。
- **ソロで守れるmoat**: ①ディストリビューション ②独自データのフィードバックループ ③ワークフロー統合の深さ（剥がすと業務が回らない） ④業界特化×規制知識（「規制の枠組みは障壁ではなく設計図」）
- **価格の算術**: $499/月なら17社で$100K ARR、84社で$500K ARR。手売り営業で足りる規模。

### レッドチームが指摘した全体への警告

1. **「ペインの確認」≠「競合の不在」**。12案中10案で名指しできる直接競合が既に出荷済みだった。ニーズ調査の出典の多くが競合ベンダー自身のマーケ記事であり、「ペインの証拠がそのまま競合実在の証拠」という構造。
2. **米国案6件は「市場の魅力」で選ばれ「自分が売れるか」で選ばれていない**。日本在住・現地ネットワークなしで、電話と信頼で回る米国中小B2Bに手売りする現実性は低い。
3. **精度・責任リスクの過小評価**。請求金額・保険証明・レセプトなど「AIの間違い＝顧客の金銭損失」に直結する領域は、人間承認フローの負荷が高いと「結局手でやるのと同じ」となり解約される。
4. **依頼主の唯一にして最大の武器は「日本の建設業の現場アクセス」**。それが活きる案に集中すべき。

## 第3部: 推奨トップ3（統合後の最終判断）

レッドチームの指摘を踏まえ、生成時の原案から再設計した推奨形。

### 🥇 推奨1位: 建設下請け向け「作業員マスターデータハブ」（安全書類パイロット改）

**何をするか**: 下請・専門工事業者が作業員の情報（資格証・保険・健康診断・入退場記録）を一度だけ登録すれば、①元請ごとに要求される安全書類（グリーンファイル・施工体制台帳）、②出面（勤怠・配置）記録、③石綿等の法定記録が全部そこから生成される「下請け側のマスターデータ基盤」。

**なぜこれか**:
- 12案で唯一、**ドメイン知識（建設実務）・既存開発資産（出面管理・石綿記録システム）・営業チャネル（業界人脈）の3点が揃う**。成功事例の共通パターン「自分の専門領域から始める」に完全一致。
- 2024年問題（残業規制）という法的ドライバーが実在し、書類AI化で年200万円相当の削減試算という支払意欲の根拠がある。
- 出面データとの統合は「剥がすと業務が回らない」ワークフロー統合型moatになる。

**競合と差別化（ここが生死を分ける）**:
- Greenfile.work（下請無料）・グリーンサイト・Buildeeが「元請への提出」を握っている。**「安全書類作成ツール」として正面から出すと無料競合に轢かれる。**
- 勝ち筋は「下請け側は複数の元請ポータルへの再入力を強いられている」という空白。どのポータルにも紙様式にも吐き出せる**下請け側のハブ**として設計し、提出先ポータルとは競合ではなく「出力先」の関係にする。

**リスク**: 単体の書類作成は模倣容易。元請ポータルにAPIがない場合、転記支援（CSV/帳票出力）止まりになる。市場規模は国内建設下請けに限定される。

**最初の一歩（90日プラン）**:
1. 開発中の出面・石綿システムに「作業員マスター→全建統一様式（労務安全書類）PDF出力＋資格証写真のAI読取」を追加（1〜2ヶ月）
2. 業界人脈から5社に有償パイロット（月1〜3万円でも「無償にしない」こと。支払意思の検証がMVPの目的）
3. パイロット中に「どの元請の・どの様式への転記が一番苦痛か」を記録し、対応様式を苦痛順に拡張

### 🥈 推奨2位: 士業向け 建設業許可・経審の「資料整理・上流」AI（許認可コパイロット改）

**何をするか**: 行政書士事務所向けに、顧客（建設会社）から届く雑多な資料（決算書・工事経歴・資格証・登記簿）をAIが整理・構造化し、不足資料を自動で督促、既存の様式作成ソフトやJCIP（電子申請）に流し込める状態にする。

**なぜこれか**: 建設業許可は5年ごと更新＋経審は毎年という継続需要で、士業は費用を顧客に転嫁できるため支払意欲が高い。依頼主の建設知識を「支払能力の高い層（士業）」に売る変換。日本語×行政手続きがグローバル大手を構造的に遠ざける。

**競合と差別化**: 様式作成ソフト（許認可プロ・マルット等）は30年前から存在し、国交省のJCIPが転記自体を電子化しつつある。**「様式作成」では戦わず、既存ソフトが扱わない上流（資料の受領・判定・不足督促）に絞って補完関係を作る。**（米国でTaxDome等が証明した「資料回収AI」パターンの日本・建設許認可版）

**リスク**: 行政書士実務の内側は未経験のため、最初の設計パートナーとなる事務所が1〜2件必要。既存ベンダーがLLM機能を足す速度との競争。

**最初の一歩**: 建設業許可「更新」1類型・1都道府県に絞ったプロトタイプを作り、知り合いの建設会社経由で行政書士1〜2名に当てて業務フローを観察する。

### 🥉 推奨3位: 建設資材卸・納材業向け FAX/納品書AIデータ化（FAXブリッジ建設特化版）

**何をするか**: 建設資材卸・納材業者に届くFAX注文書・手書き伝票をマルチモーダルAIで構造化し、販売管理システムへ投入。将来的には現場側の出面・請求データとの突合まで。

**なぜこれか**: 「受け手側だけで完結する」構造は取引先の合意が不要で営業が軽い。手書き・かすれ・独自様式の読解はマルチモーダルLLMで初めて実用化した領域。

**競合と差別化**: 汎用FAX-OCR（インフォマート・CO-NECT等）は月額1万円/1,000枚の価格アンカーが形成済みで、**汎用では絶対に戦わない**。建設資材という1業種の様式・商習慣（現場名・掛率・配達便）に特化し、依頼主の業界知識で汎用勢が読めない文脈を読む。

**リスク**: 汎用勢が建設向けテンプレートを足せば差が縮む。moatは相対的に薄く、1位・2位の案との連携（同じ建設業データ基盤に載せる）が前提。

### 番外: VerticalKit（ボイラープレート）は「売らずに作る」

推奨1〜3位はすべて「ドキュメント取込→AI抽出→確認UI→様式出力」という同型のパイプライン。この共通基盤を内部資産として最初から切り出しておくと、2つ目・3つ目の垂直展開が数週間になる。**販売はオーディエンスができた後の話**（Marc Louの$1MはTwitterオーディエンスの収益化であって製品の勝利ではない、というのがレッドチームの結論）。

## 第4部: 実行にあたっての原則（調査からの逆算）

1. **価格は月3万円（$200）以上のB2Bから下げない** — $50/月未満のAI製品はGRR 23%で構造的に死ぬ。「人件費の直接代替」でROIを語れる価格帯を守る。
2. **最初の5社は人脈で有償パイロット** — ソロ起業の死因第1位は顧客獲得。無償パイロットは支払意思を検証できないので必ず課金する。
3. **AIの間違いが顧客の金銭損失に直結する箇所には必ず人間承認を挟み、承認が数秒で終わるUIに投資する** — 「結局全部見直すなら手でやるのと同じ」と言われたら解約される。
4. **モデル進化が追い風になる位置取りを守る** — 様式知識・業界文脈・ワークフロー統合はモデルが良くなるほど製品が良くなる。汎用機能（要約・チャット）は作らない。
5. **「ペインの確認」と「競合の棚卸し」は必ず別工程で行う** — 本調査の最大の教訓。新しい垂直に出る前に、競合を名指しで10社リストアップしてから作る。
