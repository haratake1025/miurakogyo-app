## 1\. システム概要・スコープ

現場ごとの **出面管理** と **石綿作業従事者作業記録簿** を、CBOの案件・出面報告と双方向同期しながら作成・編集するPC向けWebアプリ。

**含む**: 現場・作業者マスタ・既存出面のpull、出面/石綿記録の追加・修正、CBOへのpush、同期ログ、帳票出力。 **含まない（初版）**: 給与・実行予算・請求書連携、リアルタイム同期、モバイル最適化（後日）。

---

## 2\. 対象ユーザーと権限・動作環境

- 想定端末: **PC（デスクトップ）優先**。レイアウトは広い画面のグリッド表示に最適化。モバイルは将来対応。  
- ロール【要確定の細部あり】: 管理者（全現場・同期・マスタ確認）／現場責任者・職長（担当現場の入力）。  
- アプリのログイン認証は **Supabase Auth【仮】**（CBOトークンとは別。トークンはサーバ側のみ保持）。

## 3\. CBO連携の基本方針

### 3.1 認証

三浦興業用 Sanctum トークンをサーバ側シークレットに保持し `Authorization: Bearer <token>`。**フロント露出禁止・必ずバックエンド経由**。company\_id=4083 はレスポンス検証の二重チェック。

### 3.2 同期モデル（手動・双方向）

- **「CBOから取込」(pull)**: 現場（asbests=✔）・作業者マスタ・指定期間の出面を取得しアプリDBへ反映。  
- **「CBOへ反映」(push)**: アプリで新規/修正した出面記録を送信。  
- 全同期を `sync_logs` に記録。push後は `save_async` 再計算を待って再pullし整合確認。

## 4\. データモデル

### 4.1 中核概念

**1件の「出面記録」= 出面(4879)の1レポート \= 現場 × 作業者 × 日付。** 1作業者×1日=1レコード（作業内容は単一）。出面表ビューと石綿記録ビューは同一 `daily_reports` の2表現。1セル＝1レコードに対応。

### 4.2 アプリ内部スキーマ（Supabase / Postgres）【仮】

```
sites（現場マスタ：案件2556からpull / asbests=true のみ）
  id uuid PK
  cbo_order_id        text unique     -- onsite連携キー
  name                text            -- contract_name
  client_name         text            -- suppliers_name
  manager_name        text            -- order_staff
  is_asbestos         boolean         -- asbests
  period_start date / period_end date -- start_date_man / end_date_man
  last_synced_at      timestamptz

workers（作業者マスタ：CBO→app一方向。アプリ新規登録不可）
  id uuid PK
  source_kind         text            -- 'employee' | 'partner'
  cbo_company_user_id text            -- employee: company_users.id
  cbo_supplier_id     text            -- partner: 協力会社supplier id
  cbo_supplier_staff_id text          -- partner: staff id
  company_name        text            -- employee=三浦興業 / partner=name
  worker_name         text            -- employee=姓+名 / partner=staff_last_name
  name_kana           text            -- employee: 姓カナ+名カナ
  tel                 text
  is_active           boolean
  last_synced_at      timestamptz

daily_reports（出面記録：出面4879の1レポート）
  id uuid PK
  site_id             uuid FK -> sites
  worker_id           uuid FK -> workers
  work_date           date
  reporter_id         text            -- main_reporter
  day_yakan_id        text            -- 105360/105361
  over_hour           numeric
  work_content_id     text            -- 単一（106548〜106555）
  health_type_id      text            -- 106556〜106558
  is_corrected        boolean
  cbo_report_id       text            -- CBO報告ID（null=未push）
  sync_status         text            -- 'local_new'|'synced'|'local_edited'|'conflict'
  created_by/updated_by text
  created_at/updated_at timestamptz
  unique(site_id, worker_id, work_date)   -- 1作業者×1日=1レコード

sync_logs（同期ログ：監査）
  id uuid PK / direction('pull'|'push') / target('site'|'report'|'master')
  record_id uuid / cbo_report_id text / status('success'|'error')
  message text / payload_snapshot jsonb / performed_by text / performed_at timestamptz

day_yakan_options / work_content_options / health_type_options
  {id, label, position}
```

## 5\. 画面仕様（PC優先）

### 5.1 現場一覧（トップ）

asbests=✔ の現場を表/カードで一覧（工期・現場責任者・取引先名）。各現場から「出面表」「石綿記録」へ。「CBOから取込」ボタン。

### 5.2 出面表（現場別・月次グリッド）

- 縦=作業者（所属会社＋氏名）、横=日付（曜日・祝日色分け）、月送り（◀ ▶）  
- セルで在席／昼勤夜勤／残業を入力 → daily\_reports 生成・編集（1セル=1レコード）  
- **「＋作業者を追加」**: ①会社名選択（三浦興業=社員マスタ／協力会社=該当外注staff）→ ②**氏名を複数チェックして一括追加**（既存マスタ作業者の行追加であり、新規マスタ登録ではない）

### 5.3 石綿作業従事者作業記録

- 上期（1〜15日）/ 下期（16〜末日）切替  
- 縦=作業者、横=日付、各セルに 作業内容（単一）＋健康状態（〇/△/✖）  
- 修正済セル明示（is\_corrected）  
- ヘッダ: 管轄工事会社・工事名称・工事期間・現場責任者  
- 「出面集計Excel出力」「印刷（石綿記録）」

### 5.4 マスタ管理

作業者マスタ（社員/外注）の**閲覧のみ**。追加・編集はCBO側が正。

### 5.5 同期・ログ

「CBOへ反映」実行と結果、`sync_logs` 閲覧。

## 6\. CBO連携仕様（pull / push）

### 6.1 pull

1. 案件2556（asbests=true）→ sites upsert（cbo\_order\_id基準）  
2. company\_users（自社員）／協力会社5307 supplier+staff（外注）→ workers upsert  
3. 指定現場×期間の出面4879 → daily\_reports upsert（cbo\_report\_id基準）。ローカル編集中と差分は `conflict`

### 6.2 push

- `sync_status in ('local_new','local_edited')` を対象  
- 作業者種別で振り分け: employee → `work_user`、partner → `sup_name`\+`sup_staff`  
- 新規=create（戻り report\_id を保存）、既存=cbo\_report\_id 指定で update  
- 成否を sync\_logs に記録、成功で `synced`

### 6.3 競合処理【仮】

同一 cbo\_report\_id がpull時に更新済みなら `conflict` とし、画面で比較解決。API側の更新検知（楽観ロック/updated\_at）可否は §9-2 次第。

### 6.4 CBO APIへの確認事項（着手の最小4点）

1. 出面4879 **作成の実例**（URL/メソッド＋成功JSON＋レスポンス report\_id）  
2. custom\_group値の渡し方（id か value）  
3. 外注: `sup_name` 用の会社(協力会社)id 取得経路＋`sup_staff` id  
4. 読み取り系: 案件(2556 asbests絞り)／company\_users／協力会社5307+staff／出面（現場×期間）の一覧取得
