# バックエンド設計書（v0.1）― 出面・石綿記録システム

本体仕様書 §4・§6、CBO API連携仕様 を実装に落とすためのバックエンド設計。

---

## 1\. 技術スタック

| 層 | 採用 | 理由 |
| :---- | :---- | :---- |
| ホスティング | Vercel | 既存v2と同系。サーバ関数（Route Handler）同居・無料枠 |
| アプリ | Next.js（App Router）/ TypeScript | フロント＋APIルートを1リポジトリで |
| DB | Supabase（Postgres） | 無料枠で十分。Auth同梱 |
| 認証 | Supabase Auth | アプリログイン（CBOトークンとは別） |
| CBO連携 | サーバ側 fetch（CBO API直叩き） | トークンをフロントに出さない。GASは挟まない |

**信頼境界**: フロントは秘密を持たない。`CBO_TOKEN`・Supabase `service_role` は**サーバ側のみ**。フロント → **Next.js APIルート** → (Supabase / CBO API)。

環境変数（サーバ）:

```
CBO_BASE_URL = https://office.craft-bank.com/api
CBO_TOKEN    = <三浦興業トークン>
CBO_COMPANY_ID = 4083
SUPABASE_URL / SUPABASE_SERVICE_ROLE_KEY
NEXT_PUBLIC_SUPABASE_URL / NEXT_PUBLIC_SUPABASE_ANON_KEY   # フロントAuth用のみ
```

## 2\. Supabase テーブル定義（DDL）

```sql
-- 拡張
create extension if not exists "pgcrypto";

-- ===== 選択肢マスタ（CBO custom_type） =====
create table day_yakan_options    (id text primary key, label text not null, position int);
create table work_content_options (id text primary key, label text not null, position int);
create table health_type_options  (id text primary key, label text not null, position int);

-- ===== 現場マスタ（案件2556, asbests=true のみpull） =====
create table sites (
  id uuid primary key default gen_random_uuid(),
  cbo_order_id   text unique not null,
  name           text not null,           -- contract_name
  client_name    text,                    -- suppliers_name（管轄工事会社/元請）
  manager_name   text,                    -- order_staff
  is_asbestos    boolean not null default true,
  period_start   date,                    -- start_date_man
  period_end     date,                    -- end_date_man
  last_synced_at timestamptz,
  created_at     timestamptz not null default now(),
  updated_at     timestamptz not null default now()
);

-- ===== 作業者マスタ（CBO→app一方向） =====
create type worker_source as enum ('employee','partner');
create table workers (
  id uuid primary key default gen_random_uuid(),
  source_kind            worker_source not null,
  cbo_company_user_id    text,            -- employee: company_users.id
  cbo_supplier_id        text,            -- partner: 協力会社supplier id（sup_name用）
  cbo_supplier_staff_id  text,            -- partner: staff id（sup_staff用）
  company_name           text not null,   -- employee=三浦興業 / partner=name
  worker_name            text not null,   -- employee=姓+名 / partner=staff_last_name
  name_kana              text,
  tel                    text,
  is_active              boolean not null default true,
  last_synced_at         timestamptz,
  created_at             timestamptz not null default now(),
  updated_at             timestamptz not null default now()
);
-- 重複防止（NULL混在を避けるため部分unique）
create unique index workers_employee_uq on workers (cbo_company_user_id) where source_kind='employee';
create unique index workers_partner_uq  on workers (cbo_supplier_id, cbo_supplier_staff_id) where source_kind='partner';

-- ===== 出面記録（出面4879の1レポート＝現場×作業者×日） =====
create type sync_status_t as enum ('local_new','synced','local_edited','conflict');
create table daily_reports (
  id uuid primary key default gen_random_uuid(),
  site_id            uuid not null references sites(id) on delete cascade,
  worker_id          uuid not null references workers(id),
  work_date          date not null,
  reporter_cbo_user_id text,                       -- company_user_id（報告者）
  day_yakan_id       text references day_yakan_options(id),
  over_hour          numeric not null default 0,
  work_content_id    text references work_content_options(id),
  health_type_id     text references health_type_options(id) default '106556',
  is_corrected       boolean not null default false,
  cbo_report_id      text unique,                  -- null=未push
  sync_status        sync_status_t not null default 'local_new',
  cbo_synced_at      timestamptz,                  -- 直近pull時のCBO更新基準（競合検知用）
  created_by         text,
  updated_by         text,
  created_at         timestamptz not null default now(),
  updated_at         timestamptz not null default now(),
  unique (site_id, worker_id, work_date)           -- 1作業者×1日=1レコード
);
create index daily_reports_site_date_idx on daily_reports (site_id, work_date);
create index daily_reports_sync_idx      on daily_reports (sync_status);

-- ===== 同期ログ（監査） =====
create table sync_logs (
  id uuid primary key default gen_random_uuid(),
  direction        text not null check (direction in ('pull','push')),
  target           text not null check (target in ('site','worker','report')),
  record_id        uuid,
  cbo_report_id    text,
  status           text not null check (status in ('success','error')),
  message          text,
  payload_snapshot jsonb,
  performed_by     text,
  performed_at     timestamptz not null default now()
);
create index sync_logs_performed_idx on sync_logs (performed_at desc);

-- updated_at 自動更新トリガ
create or replace function set_updated_at() returns trigger language plpgsql as $$
begin new.updated_at = now(); return new; end $$;
create trigger trg_sites_u   before update on sites         for each row execute function set_updated_at();
create trigger trg_workers_u before update on workers       for each row execute function set_updated_at();
create trigger trg_reports_u before update on daily_reports for each row execute function set_updated_at();
```

## 3\. APIルート構成（Next.js App Router / `app/api/...`）

| 用途 | メソッド/パス | 処理 |
| :---- | :---- | :---- |
| 現場一覧（アプリDB） | GET `/api/sites` | sites を返す |
| マスタ同期（pull） | POST `/api/sync/pull/masters` | CBOから 案件(2556,asbests)・company\_users・協力会社5307 を取得 → sites/workers upsert |
| 出面取込（pull） | POST `/api/sync/pull/reports` body:{siteId,from,to} | CBOの出面4879取得 → daily\_reports upsert（cbo\_report\_id基準・競合検知） |
| グリッド表示 | GET `/api/sites/[siteId]/reports?month=YYYY-MM` | daily\_reports（+worker/options join）を返す |
| 作業者一覧 | GET `/api/workers?company=...` | 一括追加モーダル用 |
| 出面 作成 | POST `/api/reports` | local\_new で1件作成 |
| 出面 更新 | PATCH `/api/reports/[id]` | 値更新。synced→local\_edited に遷移 |
| 出面 削除 | DELETE `/api/reports/[id]` | UI二重確認前提。石綿記録は慎重に（§5） |
| 反映（push） | POST `/api/sync/push` body:{siteId, ids?} | local\_new/local\_edited を CBOへ create/update。成功で synced＋cbo\_report\_id保存。全件ログ |
| 同期ログ | GET `/api/sync/logs` | sync\_logs 一覧 |

CBO連携アダプタは `lib/cbo/`（サーバ専用）に集約。読み書きは CBO API連携仕様 §6・§8 の関数群。

## 4\. 同期ステートマシン（daily\_reports.sync\_status）

```
[作成]            → local_new
local_new  + push成功 → synced（cbo_report_id 保存）
synced     + ローカル編集 → local_edited
local_edited + push成功 → synced
[pull時] CBO側が cbo_synced_at より新しく更新されていて
         かつローカルが local_edited → conflict
conflict   → UIで解決：CBO版採用（上書き取込）/ アプリ版採用（再push）
[削除]     ローカル削除 + push DELETE → 物理削除（sync_logsに記録）
```

- push は「現場単位」または「選択行」。実行後は `save_async` 再計算を待ち、必要なら再pullで整合確認。  
- レート対策: CBO呼び出しはスロットリング（連続実行で間隔を空ける）。

## 5\. 認証・権限・削除ガード

- **アプリ認証**: Supabase Auth。フロントは anon キーでログインのみ。データ操作は Next.js APIルートを経由。  
- **DBアクセス**: サーバ側ルートは Supabase `service_role` を使用（RLSをバイパス）。認可はAPIルート層で判定。  
  - 単一企業・少人数のため初版はこの方式で簡素化。将来マルチ化するならRLSを有効化（`authenticated` ロールに現場単位の行制御）。  
- **削除ガード**: 石綿作業記録は40年保存の法定書類。削除は ①UIで二重確認 ②`sync_logs` に必ず記録 ③可能なら「アプリ側は論理削除フラグ→明示操作でのみ CBO 物理DELETE」。

---

## 6\. 無料枠の見積り

- データ量: 社員〜20名＋外注、石綿現場 数十、出面 年間 数千行 → 数MB規模。Supabase無料枠（DB 500MB / 帯域5GB）に対し余裕。  
- 注意: 無料Supabaseは7日無アクティブで一時停止 → 定期利用なら実害なし。**正本はCBO**なのでデータ消失リスクは低い。
